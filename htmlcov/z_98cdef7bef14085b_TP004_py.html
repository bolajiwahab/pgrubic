8class Char(linter.BaseChecker): 

9 r"""## **What it does** 

10 Checks for usage of char. 

11 

12 ## **Why not?** 

13 Any string you insert into a char(n) field will be padded with spaces to the 

14 declared width. That's probably not what you actually want. 

15 

16 The manual says: 

17 

18 > Values of type character are physically padded with spaces to the specified 

19 > width n, and are stored and displayed that way. 

20 > However, trailing spaces are treated as semantically insignificant and disregarded 

21 > when comparing two values of type character. In collations where whitespace is 

22 > significant, this behavior can produce unexpected results; for example 

23 > SELECT 'a '::CHAR(2) collate "C" < E'a\n'::CHAR(2) returns true, even though 

24 > C locale would consider a space to be greater than a newline. 

25 > Trailing spaces are removed when converting a character value to one of the 

26 > other string types. Note that trailing spaces are semantically significant in 

27 > character varying and text values, and when using pattern matching, that is LIKE 

28 > and regular expressions. That should scare you off it. 

29 

30 The space-padding does waste space, but doesn't make operations on it any faster; 

31 in fact the reverse, thanks to the need to strip spaces in many contexts. 

32 

33 It's important to note that from a storage point of view char(n) is not a 

34 fixed-width type. The actual number of bytes varies since characters may take more 

35 than one byte, and the stored values are therefore treated as variable-length anyway 

36 (even though the space padding is included in the storage). 

37 

38 Sometimes people respond to "don't use char(n)" with "but my values must always be 

39 exactly N characters long" (e.g. country codes, hashes, or identifiers from some 

40 other system). It is still a bad idea to use char(n) even in these cases. 

41 

42 Remember, there is no performance benefit whatsoever to using char(n). 

43 In fact the reverse is true. One particular problem that comes up is that if you try 

44 and compare a char(n) field against a parameter where the driver has explicitly 

45 specified a type of text or varchar, you may be unexpectedly unable to use an index 

46 for the comparison. This can be hard to debug since it doesn't show up on 

47 manual queries. 

48 

49 ## **When should you?** 

50 When you're porting very, very old software that uses fixed width fields. Or when 

51 you read the snippet from the manual above and think "yes, that makes perfect sense 

52 and is a good match for my requirements" rather than gibbering and running away. 

53 

54 ## **Use instead:** 

55 1. text 

56 2. text with a constraint that enforces a maximum string length 

57 """
